Part 1 — What is Claude Code?

Claude Code is an AI coding assistant that runs in your terminal. You type natural language ("add a login page") and it reads your files, writes code, runs commands, and builds features — all while you watch and approve each step.

Installing & Starting

Install (one time)
npm install -g @anthropic-ai/claude-code
Start a session in any project
cd my-project claude

That's it. You're in an interactive chat with Claude inside your project. It can see your files, understand your codebase, and make changes.

Things you can say inside Claude Code
"explain what src/auth.py does" "add a logout button to the navbar" "fix the bug where users see a blank page after login" "run the tests and fix any failures"

1
Start with an Idea start here

You have something you want to build. "Add dark mode" or "let users export to PDF." Before touching any code, you need a plan.

Like an architect drawing blueprints before construction begins.
2
Design & Architecture recommended

Ask Claude Code to think through the design before building. This catches bad decisions early.

Built-in way (plan mode)
> "plan how to add dark mode to this app"

Claude enters plan mode — it explores your codebase, identifies the right files, and presents an implementation plan for your approval before writing any code.

Custom skill (Dave the Architect)
> /dave > "I want to add dark mode. What's the best approach?"

Dave is a custom "senior architect" persona that hardens specs, spots edge cases, and picks the right patterns. He asks tough questions before you commit to a design.

Why this matters: 30 minutes of design saves hours of rework. Even if you skip it for small tasks, always design for anything touching multiple files or adding new features.
3
Create a Branch required

A branch is like a parallel universe for your code. You make changes there without affecting the main version.

Create and switch to a new branch
git checkout -b feature/dark-mode

git checkout -b = create a new branch AND switch to it. feature/dark-mode = the branch name (convention: feature/, fix/, or chore/ prefix).

Like creating a "Draft" copy of a Google Doc. Edit freely — the original is untouched.
4
Build the Feature required

Now you write code. With Claude Code, describe what you want and it writes it for you, showing every change for approval.

Interactive building (you + Claude)
> "implement dark mode following the plan we made"
Headless / overnight mode (Ralph)
claude -p "implement dark mode per the spec in SPEC.md" \ --allowedTools Edit,Write,Bash,Read,Glob,Grep

Ralph = running Claude Code autonomously. Give it a spec, let it work unattended (e.g., overnight). The -p flag runs a single prompt without interactive chat.

Saving Progress (Commits)

git add src/theme.js src/components/ThemeToggle.jsx git commit -m "Add dark mode toggle component"

git add = stage files (mark for saving). git commit -m = save a snapshot with a description. Commits are "save points" — make them often.

5
Design Review recommended for UI

If your feature has a user interface, get the design reviewed before moving to QA. This catches usability problems, visual hierarchy issues, and accessibility gaps.

Design critique (Nielsen's heuristics)
> /design-critic > [paste a screenshot or describe the UI]

Evaluates your UI against established usability principles. Catches poor contrast, missing error states, confusing navigation, inconsistent spacing.

Accessibility audit
> /a11y-audit

Checks against WCAG 2.2 AA: color contrast, keyboard nav, screen readers, focus management.

Mobile-specific review
> /julie > "review the mobile experience of this page"

Julie is a mobile web UX specialist. Checks touch targets, scroll behavior, viewport issues.

Don't skip design review for user-facing features. Code that works perfectly can still be unusable. A 5-minute design check prevents "it works but nobody can figure it out."
6
QA / Testing recommended

Before sharing your work, verify it actually works end-to-end.

> /qa > "verify dark mode works across all pages"

QA Sentinel runs 6 phases: read requirements, identify scenarios, run tests, check edge cases, verify behavior, report findings.

7
Push to GitHub required

Upload your branch to GitHub so others (and review tools) can see it.

git push -u origin feature/dark-mode

Your code is now on GitHub but still NOT merged into main. It's parked there as a draft.

8
Open a Pull Request (PR) required

A Pull Request = "Here are my changes. Review them before they become official." Shows every line changed, a description, and a place for discussion.

gh pr create --title "Add dark mode" --body "Adds theme toggle..."

gh = GitHub CLI. pr create = create a pull request.

Like submitting an essay for review — "here's what I wrote, please check it before we publish."
9
Code Review recommended

Claude Code's /code-review dispatches 5 AI agents to independently review your PR: bugs, security, rule compliance, git history, code comments.

Review locally
> /code-review
Post review comments directly to GitHub PR
> /code-review --comment
Quick spot-check (lighter, single-pass)
> /richter
10
Fix & Merge required

Fix review issues, push again, then merge.

git add -A && git commit -m "Fix review findings" git push
Merge (squash = combine all commits into one clean one)
gh pr merge --squash

Squash merge = all your branch commits become one clean commit on main. Feature is now official.

11
Deploy required

Make your changes live. Depends on your project — restart a server, push to cloud, or auto-deploy on merge.

npm run build && npm start

Part 3 — Hooks (Automatic Safety Checks)

Hooks are rules in a CLAUDE.md file that Claude follows automatically. They trigger on specific events and run safety checks. Think of them as guardrails.

Two levels: global (all projects) and project-level (one repo).

Public Repo Safety Gate

Triggers: before any git add or git commit in a public repo

Scans every file for secrets, internal strategy docs, competitor analysis, or private info. Blocks the commit if anything sensitive is found.

Pre-Commit Gate

Triggers: before any git commit

Runs type checking, all tests, and a full build. If anything fails, the commit is blocked until you fix it. Prevents broken code from ever being saved.

QA Gate (Pre-Deploy)

Triggers: before any deploy, restart, or "it's ready" message

Validates types, tests, browser JS syntax, builds, and rendered HTML output. Ensures what you're about to make live actually works.

Post-Build Review Gate

Triggers: after completing changes, before saying "done"

Runs CI checklist, checks for stale TODOs, verifies test coverage for new functions, validates module boundary types.

Visual Plan Presentation

Triggers: when exiting plan mode (for plans with 3+ sections)

Generates a visual HTML page instead of plain text plans. Opens in the browser for easier review.

How to Create Your Own

Add rules to your CLAUDE.md file:

## My Custom Hook **HOOK: Before [trigger event], do [these checks]. No exceptions.** 1. Step one... 2. Step two...

Claude reads these instructions and follows them automatically in every session.

Part 5 — Key Concepts

CLAUDE.md

Instructions file Claude always follows. Global (~/.claude/CLAUDE.md) for all projects, or per-project in repo root. Like "house rules."

Branch

Isolated copy of your code. main is the real version. Feature branches are drafts. Merge when ready.

Commit

A save point. Records what changed and why. You can always go back to any previous commit.

Pull Request (PR)

Request to merge your branch into main. Shows all changes, enables discussion and review.

Plan Mode

Claude explores your codebase and designs an approach before writing code. You approve the plan first.

Headless Mode (Ralph)

Running Claude with -p flag for autonomous, unattended work. Give it a spec, let it build overnight.

Squash Merge

Combining all branch commits into one clean commit when merging. Keeps main history tidy.

Hook

Automatic rule in CLAUDE.md triggered on events (commit, deploy, plan exit). Runs safety checks without you remembering.
